# Delve AI

A new way to interact with LLMs too generate content. Delve AI focuses on two things: orchestrating LLM agents to maximise context efficiency, and surfacing artifacts generated by LLMs to humnans in an intuitive and easy to use way.

## Principles of design

Delve AI will be built around two core components:
* A CLI written in Rust -- the CLI will provide all the core capabilities for managing sessions with Delve. It will be the core of the app and should be built first. The CLI will be used to iterate on and refine the interaction model for Delve.
* A local web based front end -- this will provide an easy to use front end for Delve. Once the core interactions are refined using the CLI the local web frontend will be built on top of it.

## Core concepts within Delve

### All sessions are trees
* Every session in Delve is represent as a tree
* The initial prompt (called the Intent) is a the root node of the tree
* Every node is either a Prompt or a Artefact
* An Artefact is either a Context or an Implementation
* Context nodes are intended to be used by subsequent prompts to provide additional Context for the output of the Prompt
* Implementations are the leaf nodes of the tree that have been Accepted as correct and represent the successful output of the session
* Artefact nodes can be Accepted or Rejected
* Rejected nodes should NEVER be used as context in future prompts
* Accepted nodes CAN be used for context in future prompts
* There is always a "Current Node" until the Intent is considered complete


### State managment is via session folders, markdown and JSON
* Every Intent creates an IntentLabel that is a short version (16 characters) of the intent text and an 8 character random unique ID
* All data for an intent is stored in a directory named after the IntentLabel
* Every Prompt creates a PromptLabel that is a short version (16 characters) of the prompt text and an 8 character random unique ID
* Every Prompt creates a new directory named with the PromptLabel, all Artefacts generated from the Prompt are stored in the Prompt directory
* Every Artefact has an ArtefactLabel that is a short version (16 characters) of the data in the artefact and an 8 character random unique ID
* If the Artefact is a code change, the code change is made with a branch name of the ArtefactLabel
* In all other cases (unless explicitly directed otherwise) the data for the Artefact is saved in a file with the ArtefactLabel as markdown document with a .md extension.
* The tree the represents the session should be saved as a .JSON file
* This file needs to represents the tree structure clearly. IntentLabel, PromptLabels and ArtefactLabels will be used to as node names. Every node should reference either a file or a branch/commit pair

### CLI Flow
* CLI should have an interactive mode that allows users to create a session, specify a new Intent, see the Artefacts generated from that Intent, then create a new Prompt to continue the sessions
* CLI should have an auto-interactive mode that allows a user to enter an Intent
    * Delve will then generate the Artefacts from the Intent
    * Delve will review the Artefacts from the Intent
    * Delve will create a new Prompt to continue the work
    * Delve will ask the user if they want to review the Artefacts and continue with the new Prompt
    * This process will be repeated until Delve believes the Intent is complete or the user stops the process
* At any point the user can exit the interactive session by pressing Ctrl+C twice
* Interactive mode should also allow users to show a history of all sesssions and then continue a previous sessions
* The CLI should also have a non-interactive mode that allows users to:
    * Create a new session withn an Intent passed in on the command line, this will generate Artefacts for this Intent and then exit
    * Continue a previous session, adding a new Prompt, executring that prompt and generating Artefacts and then exiting
