## Root Cause

The `suggest_next_prompt` step calls `AmpProvider::generate()`, which runs:

```
amp --no-color threads continue <thread_id> -x "<prompt>" --stream-json
```

The `-x` flag makes this a **non-interactive** `amp threads continue` call. The `amp` CLI is exiting with status code **1**, and `run_amp_stream_json` (line 344) catches this non-zero exit and returns a `ProviderError::CommandFailed`.

## Most Likely Causes

1. **The Amp thread has reached a state where it can't be continued** — the previous artifact generation step already completed work on the thread, and the follow-up `threads continue` with the hardcoded prompt `"Based on the context available, what it is the suggested to next step to complete this Intent?"` may be hitting an Amp-side limitation (e.g., context too long, rate limit, or the thread is in a terminal state).

2. **The `-x` flag combined with `--stream-json`** — the `suggest_next_prompt` call uses `generate()` (not `generate_streaming`), but `generate()` still calls `run_amp_stream_json` which uses `--stream-json`. If `amp` exits 1, the stderr contains the actual reason but the current error display at line 70 shows `"provider Amp failed with status Some(1): <stderr>"`. **Check what stderr actually says** — it likely contains the real error (auth failure, rate limit, context length exceeded, etc.).

3. **Missing or expired auth** — if the Amp CLI requires re-authentication in non-interactive mode, it would exit 1.

## How to Investigate / Fix

- **Immediate debug**: Run the failing command manually to see the real stderr:
  ```
  amp --no-color threads continue <thread_id> -x "Based on the context available, what it is the suggested to next step to complete this Intent?" --stream-json
  ```
- **Code fix**: The error message likely already contains the stderr but may be getting truncated. Consider logging/printing the full stderr from the `ProviderError::CommandFailed` to make future debugging easier.
- **Resilience fix**: Consider making the `suggest_next_prompt` step non-fatal — if it fails, the session create/continue still succeeded; only the "next prompt suggestion" is missing. Wrapping the call in a fallback that returns a default/empty suggestion would prevent the entire command from failing.
